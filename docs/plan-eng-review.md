## Executive Summary

The proposed monolith-first Go architecture is sound for v1. The dual-mode publishing requirement (full-auto + human-review) is achievable but needs a concrete flag-based split in the queue system. The most critical technical risks are: (1) no database schema defined yet, (2) Chrome Profile cookie management plan is vague, and (3) the human-review notification path is unspecified. These are all solvable — but they need explicit design before coding starts.

## 1. Architecture Review

### 1.1 Overall Architecture: Sound

The monolith-first approach (Approach A) is correct for v1. A single `redbookc-go` binary with internal package boundaries gives you a deployment artifact that's easy to ship and a code structure that's easy to navigate.

```
cmd/server/main.go         — Gin server, dashboard, API
internal/
  engine/                  — RSS feed polling (微信公众号 RSS)
  generator/               — Claude API client + prompt templates
  publisher/               — Playwright/Chromium automation
  account/                 — Chrome Profile management, credentials
  queue/                   — SQLite-backed job queue with mode flag
  middleware/              — Gin middleware (auth, rate limiting)
  stats/                   — Posting history, token usage
  recharge/                 — Token credit system (v2)
dashboard/                  — Embedded HTML/JS (//go:embed)
```

**Verdict:** No structural changes needed. The package boundaries are correct.

### 1.2 Dual-Mode Publishing: Needs Concrete Implementation Plan

**Requirement:** Two modes must coexist:
- **全自动模式 (Full-auto):** signal → generate → auto-publish
- **人工审核模式 (Human-review):** signal → generate → show user → user approves → publish

**Current state:** The office-hours design mentions both modes but doesn't specify how the queue handles them.

**Recommended implementation:**

Add a `publish_mode` field to the queue job struct and a routing switch at dequeue time:

```go
// internal/queue/job.go
type Job struct {
    ID           string
    AccountID    string
    SignalSource string
    SignalURL    string
    GeneratedPost string
    PublishMode  string  // "auto" or "review"
    Status       string  // "pending", "generated", "approved", "published", "failed"
    CreatedAt    time.Time
    ApprovedAt   *time.Time  // nil until human approves
    PublishedAt  *time.Time
}
```

The publisher reads `PublishMode`:
- `"auto"` → publish immediately after generation
- `"review"` → pause at "generated" state, send notification (see 1.4), wait for approval

This is a minimal, clean split. One enum field, one if-statement in the publisher.

**Risk:** If this isn't locked in now, the full-auto path and human-review path will get entangled as features grow. The fix is cheap now, expensive later.

**Recommendation:** Lock in the `PublishMode` field and the status state machine before any other queue work. This is a 30-minute decision that prevents a 2-day refactor.

### 1.3 Multi-Account Support: Partially Addressed

**Requirement:** Brands + content agencies both need multi-account support. Content agencies managing 50+ client accounts have fundamentally different needs from a brand running 3 accounts.

**Current plan:** Chrome Profiles at `$HOME/.redbookc-go/profiles/` with per-profile cookie jars.

**Assessment:**

For small scale (1-10 accounts), Chrome Profiles work fine. For content agencies at scale, the current plan has gaps:

| Concern | Small Brand (1-10) | Content Agency (50+) |
|---------|-------------------|---------------------|
| Profile storage | `$HOME/.redbookc-go/profiles/` is fine | Needs a configurable root path |
| Account provisioning | Manual CLI walkthrough | Needs bulk import (CSV/API) |
| Per-account rate limiting | Config per account | Needs account groups/tiers |
| Credential rotation | Manual | Automated webhook refresh |
| Monitoring | Dashboard per account | Aggregated agency dashboard |

**For v1:** The current plan handles small scale. **Flag this as a v2 concern** explicitly so the database schema is extensible.

**Recommendation:** Add an `account_type` column to the accounts table now (`"brand"` vs `"agency"`), even if agency-specific features aren't built yet. Schema migrations on a live SQLite are painful.

### 1.4 Human-Review Notification Path: Not Specified

The human-review mode requires showing the generated post to a human and getting approval. How does the system notify the human?

Options:
- **A) Polling:** User visits dashboard, sees pending posts, clicks approve/reject. Simplest to build. Works for infrequent posting.
- **B) Webhook/Email:** System sends a notification (Feishu message, email, webhook) when a post is ready for review. More complex but actually useful for content agencies.
- **C) Both:** Polling dashboard as default, webhook as v2.

**Decision needed before coding starts.** The notification mechanism affects both the API design (does the server push or does the client poll?) and the database schema (does `pending_review` have a `notified_at` timestamp?).

**Recommendation:** Go with Option C — polling dashboard for v1, webhook/Feishu notification as a v2 feature. Document it so the schema can accommodate it.

### 1.5 Distribution Architecture: In Plan, Not Yet Built

The distribution plan (GitHub Releases, `goreleaser`, optional `brew` tap) is well-thought-out. The CI/CD pipeline needs to be part of the implementation from day one — don't defer it to "we'll add it later."

**Specific gap:** No GitHub Actions workflow exists yet. This should be the first file written after the core loop is coded, not the last.

## 2. Database Schema Assessment

**Status: NOT YET DEFINED — this is the #1 gap before coding starts.**

The design mentions SQLite but no schema is provided. Here's the minimum viable schema that covers v1 requirements:

```sql
-- accounts: Chrome Profile + credential metadata
CREATE TABLE accounts (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,           -- display name
    account_type TEXT NOT NULL,   -- "brand" or "agency"
    profile_path TEXT NOT NULL,   -- path to Chrome Profile dir
    wechat_rss_url TEXT,         -- 微信公众号 RSS URL
    publish_mode TEXT NOT NULL DEFAULT 'auto',  -- "auto" or "review"
    rate_limit_posts_per_hour INTEGER DEFAULT 2,
    claude_api_key TEXT,         -- per-account key or NULL for shared
    status TEXT NOT NULL DEFAULT 'active',  -- "active", "suspended", "banned"
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- signals: RSS feed state (last seen item to avoid re-processing)
CREATE TABLE signals (
    id TEXT PRIMARY KEY,
    account_id TEXT NOT NULL REFERENCES accounts(id),
    source_type TEXT NOT NULL,   -- "wechat_rss"
    source_url TEXT NOT NULL,
    last_item_guid TEXT,          -- most recently seen item GUID
    last_polled_at DATETIME,
    UNIQUE(account_id, source_type, source_url)
);

-- jobs: the posting queue
CREATE TABLE jobs (
    id TEXT PRIMARY KEY,
    account_id TEXT NOT NULL REFERENCES accounts(id),
    signal_id TEXT REFERENCES signals(id),
    status TEXT NOT NULL DEFAULT 'pending',  -- pending, generated, approved, published, failed
    publish_mode TEXT NOT NULL,  -- "auto" or "review" (copied from account at creation)
    signal_title TEXT,          -- original article title
    signal_url TEXT,            -- original article URL
    generated_content TEXT,      -- the generated 小红书 post
    generated_at DATETIME,
    approved_at DATETIME,
    published_at DATETIME,
    error_message TEXT,
    retry_count INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- stats: aggregate posting stats per account
CREATE TABLE stats (
    id TEXT PRIMARY KEY,
    account_id TEXT NOT NULL REFERENCES accounts(id),
    date DATE NOT NULL,
    posts_published INTEGER DEFAULT 0,
    posts_failed INTEGER DEFAULT 0,
    tokens_used INTEGER DEFAULT 0,
    UNIQUE(account_id, date)
);
```

**Key decisions baked into this schema:**

1. **`publish_mode` on jobs** — copied from account at job creation time, so changing an account's mode doesn't retroactively change in-flight jobs.
2. **`signal_id` on jobs** — enables tracing a published post back to which RSS item triggered it.
3. **`account_type`** — extensible for agency-specific grouping.
4. **`stats` table is daily aggregate** — not per-post granularity, to keep query performance sane on SQLite.
5. **`profile_path` stored explicitly** — enables accounts to live on different machines if the system scales beyond single-host.

**What's NOT in v1 schema:**
- Token credit ledger (recharge system) — too complex for v1, noted in CEO review as cut from scope
- Multi-user auth — dashboard has no auth in v1, just local access
- Webhook credentials — notification system is v2

## 3. Chrome Profile & Cookie Management

**Current plan:** Store Chrome Profiles at `$HOME/.redbookc-go/profiles/<account-name>/`.

**The real problem:** Chrome Profiles are large (hundreds of MB) and contain state that can get corrupted or half-written. The Profile also contains browser fingerprinting data that 小红书 can use to detect automation.

**Specific risks:**

1. **Profile corruption:** If Playwright crashes mid-write to the profile directory, the cookies/localStorage can end up in an inconsistent state. The account would need manual re-login.

2. **Profile copying:** If you want to "clone" a working profile to a new account, you're copying hundreds of MB. Not a blocking issue but a DX problem.

3. **Login flow is manual:** The office-hours design mentions `redbookc-go account add --name my-account` as a CLI walkthrough. This is correct — automating Chrome login via Playwright is brittle and often blocked by Google. But it means the first-run experience requires human interaction.

**Recommended improvements:**

```go
// internal/account/manager.go

type Manager struct {
    profileRoot string
    browser     *playwright.Playwright
}

// ProvisionNewAccount walks the user through Chrome login
// and saves the profile to profileRoot/<name>/
func (m *Manager) ProvisionNewAccount(name string) error {
    // 1. Launch a fresh Chromium with a temp profile directory
    // 2. Open 小红书 login page
    // 3. Wait for user to complete login manually (headful mode with visible browser)
    // 4. Copy the logged-in profile to profileRoot/<name>/
    // 5. Close temp browser
}

// GetProfilePath returns the absolute path to an account's Chrome Profile
func (m *Manager) GetProfilePath(accountID string) (string, error) {
    // Lookup account.profile_path from DB
}

// LaunchBrowserForAccount creates a Playwright browser instance
// with the account's Chrome Profile
func (m *Manager) LaunchBrowserForAccount(ctx context.Context, accountID string) (*chromium.Browser, error) {
    profilePath, err := m.GetProfilePath(accountID)
    if err != nil {
        return nil, err
    }
    return m.browser.Chromium.Launch(playwright.BrowserTypeLaunchOptions{
        UserDataDir: profilePath,  // Playwright uses this as Chrome Profile
        Headless:   true,
    })
}
```

**Critical requirement:** The headful login provisioning step (Step 2-3 above) must be a **separate binary/screen** from the main daemon. Users running this on a server (no display) need clear docs that provisioning requires a GUI environment or SSH with X11 forwarding.

**Alternative for headless servers:** The setup script should document how to run the provisioning step locally (on a Mac/Linux desktop) and copy the profile directory to the server. Don't try to automate headless login — it will fail.

## 4. Signal Engine: 微信公众号 RSS

**RSS polling is straightforward.** The main architectural question is: where does the polling loop run?

Options:
- **A) In-process goroutine:** `go engine.StartPolling(accountID, interval)` — simple, shares memory with the queue
- **B) Separate process:** `poster-server` binary polls and pushes to queue — more resilient but more complex

**For v1:** Option A is correct. The CEO review already cut the poster-server split from v1 scope. Keep polling in-process until there's a concrete reason to split it.

**RSS parsing edge cases to handle:**

1. **RSS feed format variations:** 微信公众号 RSS feeds aren't always standard RSS — some are Atom, some are custom XML. Use a tolerant parser (e.g., `github.com/mmcdole/gofeed`) that handles both.

2. **Duplicate detection:** The same article can appear in multiple RSS updates if the feed doesn't use GUIDs properly. The `last_item_guid` in the signals table handles this — but only if the parser actually extracts a unique ID.

3. **Rate limiting:** RSS feeds can 403 or timeout. The polling loop needs exponential backoff with a cap (e.g., 5min → 10min → 20min → 1hr).

4. **Item deduplication:** Same article title + URL = same item. Store both `guid` and `link` in the signal state.

## 5. Claude Generator: Prompt Strategy

The highest-leverage piece of this entire system is the prompt. Bad prompts = bad posts = wasted tokens and potentially brand damage.

**Minimum prompt structure needed for v1:**

```go
// internal/generator/prompts.go

const PostTemplate = `你是一个专业的小红书内容创作者。

信号源文章：
标题：{{.SignalTitle}}
链接：{{.SignalURL}}

要求：
- 写一篇吸引人的小红书帖子，内容围绕上述文章的核心观点
- 使用表情符号增加可读性
- 每段用空行分隔
- 结尾加上相关话题标签（#话题1 #话题2 #话题3）
- 总字数控制在300-500字
- 语言风格：亲切、有共鸣感、像朋友推荐

生成的小红书帖子：
`

// internal/generator/generator.go
func (g *Generator) GeneratePost(ctx context.Context, signalTitle, signalURL string) (string, error) {
    prompt := renderTemplate(PostTemplate, map[string]string{
        "SignalTitle": signalTitle,
        "SignalURL":   signalURL,
    })
    resp, err := g.claude.Complete(ctx, &claude.CompletionRequest{
        Model:    "claude-sonnet-4-20250514",
        MaxTokens: 1024,
        Messages: []claude.Message{{Role: "user", Content: prompt}},
    })
    return resp.Content, err
}
```

**Token cost estimate:** ~500 tokens input (signal article context) + ~300 tokens output = ~800 tokens per post. At Claude Sonnet pricing, that's roughly $0.004 per post. Negligible at small scale.

**What to build as v1, not v0:**
- Prompt template versioning (store templates in DB, not hardcoded)
- A/B prompt testing (generate 2 variants, pick the better one — requires engagement data)
- Image attachment strategy (download HN cover images? Use AI image generation? No images in v1?)

**For v1:** Hardcoded prompt template with the structure above. Document it clearly so it's easy to tweak.

## 6. Playwright Publisher: The Highest-Risk Component

**This is the component most likely to break in production.**

小红书 has no official API. The Playwright flow is inherently fragile because page structure changes are:
1. **Inevitable** — 小红书 updates their UI regularly
2. **Silent** — no error, no warning, just a selector that stops matching
3. **Destructive** — a failed publish can leave the post in a half-submitted state

**The Playwright flow for publishing:**

```
1. Launch Chromium with account's Chrome Profile (headless)
2. Navigate to https://creator.xiaohongshu.com/publish/publish
3. Wait for page load, detect the "发布" button
4. Inject the generated post text into the content editor
5. Click the image upload area (if using images)
6. Handle any "detected automation" challenges (captcha, phone verification)
7. Click "发布" (Publish)
8. Wait for success confirmation
9. Record success in jobs table
10. Close browser
```

**Failure points and mitigations:**

| Failure | Detection | Mitigation |
|---------|-----------|------------|
| Page structure changed | Selector fails | Log exact step that failed, screenshot page state |
| Captcha / phone verification | Unexpected modal appears | Flag account as `suspended`, notify user, skip job |
| Network timeout during upload | Context deadline exceeded | Retry with exponential backoff (max 3 retries) |
| "Detected automation" | Page shows anti-bot message | Pause account, notify user, require manual review |
| Already published (duplicate) | Check for existing same-content post | Skip if found, log as duplicate |
| Session expired | Navigate to login page instead of creator center | Trigger re-provisioning flow |

**The most important logging statement in this entire system:**

```go
// Always log the exact step and take a screenshot on failure
logger.Error("publisher: step 4 failed",
    "step", "inject_content",
    "account_id", accountID,
    "job_id", jobID,
)
// Screenshot saved to uploads/<job_id>-failure.png
browser.Page().Screenshot(page.ScreenshotOptions{FullPage: true, Path: fmt.Sprintf("uploads/%s-failure.png", jobID)})
```

**Without this screenshot, debugging a failed publish in production is nearly impossible.**

## 7. Testing Strategy

**The testing challenge:** The publisher uses real Chromium against a real website. You cannot write a true integration test without network access to 小红书 and valid account credentials.

**Recommended test strategy:**

### Layer 1: Unit Tests (always, no excuses)

```bash
internal/
  engine/
    engine_test.go       — RSS parsing with mocked HTTP responses
    fixtures.go          — sample RSS feeds (stored as testdata/)
  generator/
    generator_test.go   — prompt rendering, response parsing
  queue/
    queue_test.go       — job lifecycle state machine
  account/
    account_test.go     — profile path resolution, validation
  stats/
    stats_test.go       — aggregation logic
```

### Layer 2: HTTP/Mock Integration Tests

```bash
internal/
  publisher/
    publisher_test.go   — mock Playwright, test the orchestration logic
                          (can test the loop without a real browser)
  engine/
    engine_integration_test.go  — against a real RSS server (testdata/httptest)
```

### Layer 3: Manual E2E (not automated)

The full publish flow needs a manual test script:

```bash
# test-e2e.sh — run manually when testing
# 1. Start with a fresh test account
# 2. Seed a known RSS item
# 3. Trigger a job manually
# 4. Verify the post appears on 小红书
# 5. Take a screenshot of the published post
```

**Coverage target for v1:**
- Unit tests: 80%+ coverage on `engine/`, `generator/`, `queue/`
- Mock integration: publisher workflow (without real browser)
- Manual E2E: core publish flow, account suspension recovery

**What's NOT testable automatically:**
- Login flow (requires human interaction)
- Captcha/verification challenges
- Page structure changes on 小红书
- Chrome Profile corruption recovery

## 8. Technical Risks

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| 小红书 updates UI, breaks selectors | CRITICAL | HIGH | Screenshot on every failure; monitor error rate; fast re-deployment |
| Account banned during automated posting | CRITICAL | MEDIUM | Human-review mode reduces ban risk; account rotation; rate limiting per account |
| Chrome Profile corruption | HIGH | MEDIUM | Keep profiles on durable storage; clear instructions for re-provisioning |
| Claude generates low-quality/offensive content | HIGH | LOW | Content filter on output; human-review mode catches before publish |
| RSS feed returns non-article content | MEDIUM | LOW | Input validation on signal data; skip items without title+URL |
| SQLite contention under concurrent access | MEDIUM | LOW | SQLite WAL mode; connection pooling; avoid long transactions |
| Token cost runaway (infinite retry loop) | MEDIUM | LOW | Max retry count per job; circuit breaker on Claude API errors |
| Data loss on crash during publish | HIGH | LOW | Job state persisted before each step; idempotent job processing |

**Top 3 actions to reduce risk immediately after coding starts:**

1. **Add structured logging with context** (`account_id`, `job_id`, `step`) — enables post-mortem debugging without guesswork
2. **Screenshot on every publish failure** — the difference between 1-hour debug and 1-day debug
3. **Rate limit with jitter** per account — `delay = baseInterval + random(0, baseInterval/2)` prevents predictable posting patterns that look bot-like

## 9. API Design (Dashboard)

The dashboard API should be minimal for v1:

```
GET  /api/accounts              — list all accounts
POST /api/accounts              — create account (triggers provisioning flow)
GET  /api/accounts/:id         — account details + recent stats
PUT  /api/accounts/:id          — update account settings (publish_mode, rate_limit)

GET  /api/accounts/:id/jobs     — list jobs for account (paginated)
POST /api/accounts/:id/jobs      — manually trigger a job (force a signal poll)
GET  /api/jobs/:id              — job detail

POST /api/jobs/:id/approve      — approve a pending-review job (human-review mode)
POST /api/jobs/:id/reject        — reject a pending-review job

GET  /api/stats                 — aggregate stats across all accounts
```

**Auth:** No auth for v1 — this assumes the dashboard is accessed locally. For production use, add a simple Bearer token middleware.

## 12. Implementation Priority Order

The order matters. Build in this sequence to avoid rework:

1. **Database schema + migrations** (`pkg/database/`) — define schema first, all other code depends on it
2. **RSS signal engine** (`internal/engine/`) — verify it correctly polls a real RSS feed
3. **Queue job system** (`internal/queue/`) — job lifecycle with `publish_mode` flag
4. **Claude generator** (`internal/generator/`) — verify prompt quality with real signals
5. **Chrome Profile manager** (`internal/account/`) — profile provisioning and browser launch
6. **Playwright publisher** (`internal/publisher/`) — the hardest piece, build and test with screenshots
7. **Gin API endpoints** (`cmd/server/`) — wire up queue to HTTP
8. **Dashboard HTML/JS** (`dashboard/`) — embedded, served by Gin
9. **GitHub Actions CI/CD** — build + test on every push
10. **Rate limiting + retry logic** — per-account backoff with jitter
