## Problem Statement

Building an AI agent that automates the full pipeline from signal ingestion to published 小红书 (Xiaohongshu/RED) content. The system monitors RSS feeds and HackerNews for interesting content, uses Claude API to generate engaging Chinese social media posts, and publishes them automatically via Playwright browser automation. Each account runs in an isolated Chrome Profile to avoid detection.

The original project brettinhere/redbookc (Python/Codong) exists. RedBookC-Go is a rewrite in Go targeting better performance, cleaner architecture, and production reliability.

## What Makes This Cool

The pipeline is end-to-end automatic: signal sources feed a content queue, Claude generates copy, and Playwright posts it — zero manual intervention after setup. Multi-account management via isolated Chrome Profiles means you can run 5 accounts simultaneously without cross-contamination. The Web Dashboard gives you live visibility into credits, posting history, and queue depth.

The "whoa" factor is watching a bot read HN, generate a thoughtful Chinese post, open Chrome, log into 小红书, and publish — without you touching anything.

## Constraints

- **Language:** Go 1.21+
- **Web framework:** Gin
- **Database:** SQLite (no external DB dependency — self-contained)
- **Browser automation:** Playwright (Chromium only)
- **AI:** Claude API (Anthropic) — cost is per-token, so prompt design matters
- **Distribution:** Single binary + static dashboard assets, easy to deploy
- **Platform:** Linux-first (Playwright headless works well on servers)
- **No official 小红书 API** — this is why Playwright is required; no workaround

## Premises

1. **小红书 posting via browser is viable** — the original redbookc proves this pattern works. Playwright can log in, navigate, fill forms, and click publish.
2. **Claude can generate engaging Chinese social copy** — with the right prompt engineering, Claude 3/4 can produce natural Chinese content that fits 小红书 style (emoji-heavy, line breaks, relatable tone).
3. **Signal-to-post is the core loop** — everything else (dashboard, accounts, credits) is support infrastructure.
4. **Isolated Chrome Profiles are necessary** — 小红书 detects bot behavior; profile isolation reduces cross-account fingerprinting.
5. **SQLite is sufficient** — at single-machine scale, SQLite handles the load (sub-1000 accounts, posting frequency of minutes-hours).

## Approaches Considered

### Approach A: Monolith First (Recommended)
Build the entire pipeline in one Go service. All components (engine, publisher, dashboard, account manager) in a single binary with internal package boundaries.

**Summary:** One `redbookc-go` binary that starts the web server, background worker, and Playwright process manager all at once.

**Effort:** M
**Risk:** Low
**Pros:**
- Single deployment artifact — `docker build` or `go build` and done
- Shared database transaction for dashboard consistency
- Easier to coordinate between the queue worker and publisher
- goroutines + channels for in-process concurrency

**Cons:**
- The poster-server (long-running browser process) might want to be separate
- Dashboard might need its own scaling axis someday

**Reuses:** internal/engine, internal/publisher, internal/account — all internal packages communicate via interfaces

---

### Approach B: Split Poster Server
Separate the long-running Playwright process from the main web/Gin service. The main service handles the API/dashboard; a dedicated `poster-server` binary manages Chrome/Playwright instances.

**Summary:** Two binaries — `redbookc-go server` (Gin API + dashboard) and `poster-server` (Playwright process manager). They communicate over HTTP or a named pipe.

**Effort:** L
**Risk:** Med
**Pros:**
- Isolated failure domains — if Playwright crashes, the API stays up
- The poster server can be restarted independently
- Cleaner separation of concerns

**Cons:**
- Two deployment artifacts instead of one
- Inter-process communication adds latency and complexity
- Must manage two processes in production
- More complex local dev setup

**Reuses:** internal/publisher moves to poster-server; internal/engine stays in main service

---

### Approach C: Cloud-Native / Microservices
Break into three services: signal engine (polls RSS/HN), content generator (calls Claude), publisher (Playwright). Each scales independently.

**Summary:** Three Go services + a message queue (NATS or Redis Streams) coordinating them.

**Effort:** XL
**Risk:** High
**Pros:**
- Each component scales independently
- Best fault isolation
- Natural team boundaries if multiple people work on it

**Cons:**
- Massive complexity for a project that doesn't need it yet
- Three deployables instead of one
- Queue infrastructure adds operational overhead
- Over-engineering for a solo/small-team project

**Reuses:** None — full rewrite

## Recommended Approach

**Approach A (Monolith First)** — ship the complete pipeline in one binary.

Start with the core loop: fetch signal → generate post → publish. The dashboard and multi-account management are v1 features but the monorepo structure makes adding them straightforward. The poster-server split (Approach B) can happen when the Playwright process needs its own lifecycle management — but that day is not today.

The architecture looks like:

```
cmd/server/main.go         — Gin server, dashboard, API endpoints
cmd/poster/main.go        — (optional v2) standalone Playwright runner
internal/
  engine/                 — signal fetching: RSS parser, HN scraper
  generator/              — Claude API client, prompt templates
  publisher/              — Playwright controller, posting workflow
  account/                — Chrome Profile management, cookie jars
  queue/                  — in-memory or SQLite-backed job queue
  middleware/             — Gin middleware (auth, rate limit, logging)
  stats/                  — posting stats, token usage tracking
  recharge/               — token credit system
dashboard/                — embedded HTML/JS for the web UI
```

## Open Questions

1. **Chrome Profile management** — Where do profiles live on disk? How does the user provision new accounts? The initial version needs a simple CLI command like `redbookc-go account add --name my-account` that walks through Chrome login.

2. **Claude prompt design** — What's the prompt structure? Topics (HN/rss), tone, format (emoji, line breaks, hashtags). This is the highest-leverage piece — bad prompts = bad posts. Should spend time building a prompt library.

3. **Rate limiting per platform** — 小红书 will throttle or ban accounts that post too fast. What's the right default delay between posts? Configurable per account?

4. **HN scraping approach** — The official HN API (Algolia) is the safest bet. `https://hn.algolia.com/api/v1/search?query=golang&sortBy=popularity`. No scraping required.

5. **Token credit billing model** — Each Claude call costs tokens. Does each user have their own Claude key, or does the system have a central pool and deduct per post?

6. **Image strategy** — Does Claude generate images (via image generation API)? Or does the system download HN cover images and attach them? Or no images in v1?

7. **What triggers posting** — Time-based (every N minutes, check queue and post one) or event-based (when new signal arrives)?

## Success Criteria

- A single `redbookc-go run` starts the dashboard at `:8080` with the full pipeline running
- The dashboard shows: account list, credit balance, posting history, queue depth
- One HN story can be manually queued and posted to a test account end-to-end
- No manual intervention required after clicking "Queue this post"
- The Playwright flow completes in under 60 seconds per post (login → new post → paste → publish → done)

## Distribution Plan

- **Binary:** `go build -o redbookc-go ./cmd/server` — single static binary, no external runtime
- **Dashboard assets:** embed via `//go:embed` — no separate `public/` directory to manage
- **Chrome Profiles:** stored in `$HOME/.redbookc-go/profiles/` — gitignored, user-provisioned
- **Database:** SQLite at `$HOME/.redbookc-go/data.db` — created on first run
- **Installer:** `brew install` tap or `docker run` for users who want one-liner deployment
- **GitHub Releases:** pre-built binaries for Linux amd64 + Darwin arm64

**CI/CD:** GitHub Actions on tag push:
1. `go build` + unit tests
2. Build cross-platform binaries (`goreleaser`)
3. Attach to GitHub Release
4. Optional: trigger Docker Hub build

## Next Steps

1. **Write the signal engine first** (`internal/engine/`). Test it independently — can it fetch HN and RSS? Does it correctly parse and store items?
2. **Build the Claude generator** (`internal/generator/`). Write prompt templates. Test against real HN stories — do the outputs look like real 小红书 posts?
3. **Implement the Playwright publisher** (`internal/publisher/`). Start with a script that opens Chrome, navigates to 小红书 creator center, and verifies the page structure. This is the highest-risk piece.
4. **Wire up the queue** (`internal/queue/`). In-memory queue is fine for v1. Add SQLite persistence when jobs start getting lost.
5. **Build the dashboard** (`dashboard/`). Start with the API endpoints, then the HTML/JS frontend. Use vanilla JS — no React/Vue overhead for a solo project.
6. **Add account management** (`internal/account/`). CLI to add Chrome Profile accounts. Cookie/credential management per profile.

## What I Noticed

This project has clear scope boundaries — the team knows exactly what they're building and why. The rewrite from Python to Go suggests they ran into performance or reliability issues with the original. The signal-to-post pipeline is well-scoped for a weekend hack: three distinct components that can be built and tested independently. The Chrome Profile isolation detail shows real thinking about platform risk — this isn't their first time dealing with social platform anti-bot measures.
